# M006 A5 — Fail-closed GitHub identity gate

`scripts/identity-assert.sh` is the control gate that makes "no autonomous
GitHub write as the human owner" enforceable. An autonomous job calls it
**before** any GitHub write; a non-zero exit means the job must not write.

## Why this exists

M006 A1 measured that autonomous sessions reach GitHub as `jayleekr` — Jay's
over-scoped admin principal. The dedicated automation principal
(`hypeproof-automation`) does not exist yet (Jay must create it — human step).
Until it does, the live cron fleet still runs as `jayleekr`. This gate is wired
now but only *bites* once Jay flips it on, so the fleet keeps working meanwhile.

## Two modes (selected by `HYPEPROOF_AUTOMATION_ACTOR`)

| Mode | Trigger | Behavior | Exit |
|------|---------|----------|------|
| **warn** (current default) | env unset/empty | resolve actual principal, log "automation identity not provisioned, running as `<actual>`", **do not block** | 0 |
| **enforce** | `HYPEPROOF_AUTOMATION_ACTOR` set | actual principal must equal it, else **fail closed**; identity unknown (gh missing/unauth) also fails closed | 0 match / 1 mismatch |

The active principal is the real write-path identity, resolved with
`gh api user --jq .login`. The token value is never printed — only the login.

**Control invariant:** every run self-reports `expected`, `actual`, `mode`,
`decision` on stdout (and stderr), so a green run proves what it checked.

## Turning on enforcement (Jay, after provisioning the token)

Once `hypeproof-automation` + its fine-grained PAT exist and cron uses that
token, export one line in the cron environment:

```sh
export HYPEPROOF_AUTOMATION_ACTOR=hypeproof-automation
```

From that point any job that is still running as `jayleekr` (or any non-matching
principal) exits non-zero and must abort before writing. That single env line is
the entire enforce switch — no code change.

## Wiring point (where jobs must call it)

The autonomous cron entrypoint is **not tracked in this harness repo** — it
lives in the hypeprooflab runtime at
`cron-prompts/run-job.sh` (see the launchd `ProgramArguments` generated by the
M005 identity-isolation package:
`source .../cron-prompts/cron-env.sh && .../cron-prompts/run-job.sh <job>`).

Add the gate as the first real step of `run-job.sh`, after env sourcing and
before any GitHub work:

```sh
# fail closed on identity before doing any GitHub write
"$HARNESS_DIR/scripts/identity-assert.sh" || {
  echo "identity gate refused GitHub writes; aborting job" >&2
  exit 1
}
```

In warn mode this never aborts (env unset), so wiring it in is safe for the
live fleet today; it starts enforcing the moment `HYPEPROOF_AUTOMATION_ACTOR`
is exported. Because that entrypoint is in another repo, wiring is documented
here rather than applied cross-repo in this PR.

## Tests

`tests/security/test_identity_assert.py` (hermetic — PATH-injects a `gh` stub):

- enforce + actual == expected → exit 0
- enforce + actual != expected → exit 1 (fail closed)
- warn + unset → exit 0 + warning naming the actual principal
- enforce + identity unknown (gh unauth / gh missing) → exit 1
- output never contains a token value
- **mutation control:** copies the guard, flips its default-deny to `pass`, and
  shows the enforce-mismatch case (exit 1 on the real guard) now exits 0 —
  proving the fail-closed exit is caused by the guard's decision, not incidental.
